Upbit Watch

A dark control panel that detects new Upbit listings the moment they appear and pings your Telegram. One FastAPI process supervises two independent detection loops plus a price-snapshot logger, streaming everything live to a browser dashboard over WebSocket. Built to run on WSL2 Ubuntu for $0 — deployable unchanged to a Seoul VPS later. The dashboard is now fully user-configurable — detector behaviour, alerts, appearance, and display — with light / dark / system themes and accent colours.

FastAPI + WebSocket Loop A markets · 1s Loop B announcements · 8s Phase 0 price snapshots SQLite WAL · survives restarts Telegram alerts Light / Dark theming + accents Fully configurable · hybrid persistence

01 Overview

What Upbit Watch is and how its pieces fit together.

Upbit Watch is one FastAPI process supervising two independent async detection loops, with everything pushed live to a browser dashboard over a WebSocket. Two design facts make it reliable:

  • Seed-silent + dedup. On first run each loop records everything that already exists and sends nothing. After that it alerts only on genuinely new items. The "already seen" set lives in SQLite, so it survives restarts — no false alert storm after a reboot.
  • One supervisor, two loops. Start / Stop / Restart (and autostart) control both loops together; each loop is an independent asyncio.Task whose every poll is wrapped in try/except, so a network blip can never crash the process.

02 Prerequisites

What you need before installing.

  • WSL2 Ubuntu (24.04) with Python 3.12 — verify with python3 --version.
  • A Telegram bot token + chat ID (see §4).
  • Outbound internet only — no port-forwarding or firewall changes. Upbit + Bybit must be reachable; Binance is optional (it's intermittently geo-blocked and treated as best-effort).
💡

Why WSL2 and not native Windows Python? Clean Linux async networking, and it's the same OS as the eventual Seoul VPS — so the code you build locally deploys unchanged.

03 Install & setup

Get the code into WSL and build the virtual environment.

1 · Project directory

bash
mkdir -p ~/upbit-bot && cd ~/upbit-bot   # already created on this machine

2 · Virtual environment (with the pip bootstrap)

On this machine python3.12-venv/ensurepip is not installed and sudo needs a password — so the venv is created without pip, then pip is bootstrapped manually:

bash
python3 -m venv --without-pip .venv
curl -sSL https://bootstrap.pypa.io/get-pip.py -o /tmp/get-pip.py
.venv/bin/python /tmp/get-pip.py
✅

If you ever have sudo access, the simpler path is sudo apt install -y python3.12-venv python3-pip then a normal python3 -m venv .venv.

3 · Dependencies

bash
.venv/bin/python -m pip install -r requirements.txt
# installs: fastapi, uvicorn[standard], aiohttp, python-dotenv
⚠️

Always invoke Python/pip as .venv/bin/python -m … — because the venv was bootstrapped manually, calling a bare pip may hit the wrong interpreter.

04 Telegram setup

Two values — a bot token and your chat ID.

  1. In Telegram, message @BotFather → /newbot → copy the token (123456:ABC...).
  2. Send your new bot any message, then open https://api.telegram.org/bot<TOKEN>/getUpdates and copy your numeric chat id.
  3. Put them in ~/upbit-bot/.env:
.env
TELEGRAM_BOT_TOKEN=123456:ABC...
TELEGRAM_CHAT_ID=987654321

…or just paste them into the dashboard Settings → Telegram and click Save. Verify with Settings → Send test message — you should get a ✅ instantly.

🔒

.env is git-ignored — never commit it. The Settings GET endpoint never returns the token, only whether it's configured.

05 Running it

Start the server, then open the dashboard in your browser.

bash
cd ~/upbit-bot
tmux new -s upbit ./run.sh        # starts uvicorn on 127.0.0.1:8000
# detach:   Ctrl-b then d
# reattach: tmux attach -t upbit
# stop:     tmux kill-session -t upbit

The raw command behind run.sh is:

bash
.venv/bin/python -m uvicorn app:app --host 127.0.0.1 --port 8000

Then open http://localhost:8000 in your Windows browser — WSL2 forwards localhost. Press Start: the first run seeds existing markets silently; afterward it alerts only on genuinely new ones.

Autostart

With "autostart": true in config.json (the default), both detectors arm themselves the instant the app launches — so after a reboot you just relaunch and it's already watching. Set it to false to press Start yourself.

Reset to a clean slate

bash
rm state.db*        # the * also removes the WAL sidecar files
# next Start re-seeds silently from scratch
⚠️

Laptop sleep/hibernate kills WSL and the bot. For long Phase 0 runs: plug in and set Windows power to "never sleep." If keeping it awake for weeks is impractical, that's the cue to move to a free Oracle Cloud Seoul box.

The core data flow — "a new coin is detected"

This is the alert fan-out, the heart of the system. Everything is decoupled through the EventBus: detectors publish() events, the WebSocket endpoint subscribe()s and forwards them — detectors never know about the browser.

new market / announcement detected
├▶
write to SQLite (seen/listings or notices) dedup memory, survives restart
├▶
engine.log(…) → EventBus "log" → dashboard Logs console
├▶
EventBus "listing"/"notice" → dashboard feed + toast
├▶
Telegram.send(…) → your phone
└▶
Phase0.schedule(market) Loop A only → snapshots at +0/10/30/60s/5m

8.1 · Loop A — market detection (detector.py → DetectorEngine)

  • _run() opens one aiohttp session and loops every poll_interval (default 1s): GET market/all, measure latency, count markets.
  • First ever run (empty seen table): bulk-insert every current market and log "Seeded N markets silently" — no alerts.
  • Afterwards: new = current_markets − seen. For each new market call _handle_new().
  • _handle_new() = the fan-out: insert into seen + listings, log an alert, publish a listing event, send the Telegram alert, and phase0.schedule(market).
  • Every poll calls publish_status() so the dashboard's uptime/latency/markets tick live. The whole poll body is wrapped in try/except — a blip increments error_count and is logged, but the loop keeps going.
  • simulate_listing() is a dev hook (/api/simulate) that forces a fake SIM-BTC through the pipeline (mapped to BTCUSDT so Phase 0 gets real prices).

8.2 · Loop B — announcement detection (notice.py → NoticePoller)

  • Polls the announcements API every poll_interval_notice (default 8s) with a browser User-Agent (the endpoint is behind Cloudflare). Dedup key is the announcement id; same seed-silent-then-alert logic, stored in notices.
  • parse_notice(title) classifies:
    • Ticker = first (UPPERCASE) group, e.g. 에스피엑스6900(SPX) … → SPX.
    • is_listing = has a ticker and the title contains a listing phrase (거래지원, 디지털 자산 추가, 신규 상장, 마켓 추가) and does not contain 종료/폐지/유의 (delisting/caution). So OXT 거래지원 종료 and KERNEL 거래 유의 are correctly excluded.
  • _handle() publishes a notice event and, only for listings, sends a Telegram alert with a link. Cloudflare 429s are caught, counted, logged — Loop A stays your reliable signal. Loop B usually fires before Loop A.

8.3 · Phase 0 — price snapshots (phase0.py → Phase0)

  • schedule(market) derives the symbol: KRW-XXX → base XXX → XXXUSDT, and spawns an async task.
  • _collect() waits to each offset +0 / 10 / 30 / 60 s / 5 m and fetches the price from Bybit (primary) and Binance (best-effort), writing both to snapshots and publishing a snapshot event for the live chart.
  • If a source fails (e.g. Binance geo-blocked), that value is stored NULL — no crash, the chart just shows a gap. Purpose: measure how big/fast the post-listing move is for your latency.

8.4 · EventBus + WebSocket (detector.py, app.py /ws)

  • EventBus is a tiny pub/sub: subscribe() returns an asyncio.Queue; publish(event) does a non-blocking put_nowait into every subscriber's queue (drops if a slow client's queue is full, so one stuck browser can't stall the detectors).
  • /ws accepts the socket, subscribes, sends an initial status, then forwards every event until the client disconnects. Event types: status, log, listing, notice, snapshot.

8.5 · Telegram (notify.py → Telegram)

  • send(text) POSTs to the raw Bot API sendMessage via aiohttp. configured() guards missing creds (returns an error dict instead of throwing). Token/chat-id are mutable at runtime so the Settings drawer can update them without a restart. /api/telegram/test sends a one-off message.

8.6 · Control & lifecycle (app.py lifespan, start/stop/restart)

  • start() launches Loop A and notice_poller.start(); stop() cancels both cleanly; restart() = stop then start.
  • Autostart: the FastAPI lifespan hook calls engine.start() on boot when config.autostart is true and engine.stop() on shutdown — a reboot + relaunch comes up already watching.

8.7 · Persistence & dedup (state.db, WAL)

  • The DB opens in WAL mode with a 3s busy_timeout, so the three concurrent writers (Loop A 1/s, Loop B, Phase 0) never hit "database is locked." Because dedup lives on disk, restarts don't re-alert — verified against 752 existing markets.

12 Troubleshooting

The common situations and what to do.

Loop B error count climbing / HTTP 429

The announcement API is Cloudflare-fronted and rate-limits residential IPs. Raise the Loop B poll interval in Settings (e.g. 10–15s). Loop A is unaffected and remains your reliable signal.

Phase 0 Binance values are blank

Binance is intermittently blocked from this network; Bybit values still fill in. This is expected and non-fatal — the chart just shows a gap.

Browser can't reach localhost:8000

Make sure uvicorn is still running (tmux attach -t upbit) and that it bound to 127.0.0.1:8000.

Bot went quiet after closing the laptop

WSL was suspended on sleep. Relaunch — autostart re-arms it.

Want a clean slate

rm state.db* (the * also removes WAL sidecar files); the next Start re-seeds silently from scratch.

🧰

Resilience at a glance: network blips are logged + counted and the loop continues; a slow browser's event queue drops overflow so detectors never block; on restart the WAL dedup means no re-seed and no false alerts.

13 Going live (VPS)

When and how to leave localhost.

Move to a Seoul VM when you start acting on alerts for real or wire up auto-trade. Almost nothing changes — the same WSL/Ubuntu code copies straight to an Oracle Cloud Seoul Always-Free ARM box (or a ~$5–12/mo Tokyo/Seoul VPS).

What you add:

  • A systemd service instead of tmux.
  • A keep-alive cron (Oracle reclaims idle instances after ~7 days).
  • A static IP for exchange-key whitelisting.
  • Trade-only API keys (no withdrawal).
🚀

Building on WSL2 Ubuntu now means zero rewrite later — that's the whole point.
